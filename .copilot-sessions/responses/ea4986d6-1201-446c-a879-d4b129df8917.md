## 22. (2026-06-30T17:03:19.492Z)

Fixed. The 1D/1W/1M/etc. timeframe switches in the Analyse tab were flaky because of a render-timing race.

## Root cause
In [web/src/components/CandleChart.tsx](web/src/components/CandleChart.tsx), the resampled candles flow through an async `setDisplayCandles` effect, so the `timeframe` prop changes **one render before** `displayCandles` updates. The view-reset effect was keyed on `[timeframe, applyDefaultView]`, so on a timeframe switch it called `applyDefaultView()` immediately against **stale** candle data (wrong length). It set a visible logical range that pointed into the middle of the new series — which is why switching to 1D sometimes showed an old window instead of the most recent bars.

## Fix
Replaced the single racy effect with a pending-reset flag:
- One effect marks `pendingResetRef = true` when `timeframe` or `symbol` changes.
- A second effect, keyed on `[displayCandles, applyDefaultView, replay.active]`, performs the reset only *after* the new data has committed (and skips during replay).

Because it now triggers off the actual data commit (which runs after the `setData` effect), `applyDefaultView()` always reads the correct candle length. Bonus: it no longer resets your zoom when toggling Heikin-Ashi or indicators.
